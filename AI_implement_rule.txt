================================================================================
매아리(MAEARI) 서비스 AI 개발 및 프론트엔드/백엔드 협업 원칙 가이드라인
================================================================================
본 문서는 매아리 서비스의 고품질 아키텍처 유지와 엔지니어 간의 긴밀한 병렬 개발을 위해
인공지능(AI) 활용 제한 조건, 무결점 개발 원칙, 모노레포 구조하에서의 명확한 분업 
및 의사결정 프로세스를 총체적으로 정의합니다.

--------------------------------------------------------------------------------
1. 무결점 개발 원칙 (Zero-Fake & Anti-Hardcoding Principles)
--------------------------------------------------------------------------------
본 프로젝트는 코드의 실제 동작 안전성을 확보하고 잠재적인 휴먼 에러를 방지하기 위해 
어떠한 형태의 가짜 구현이나 임시 하드코딩도 전면 금지하며, 아래 8대 규칙을 절대 준수합니다.

  1) [Mock 파일 생성 금지]: 테스트 목적의 임시 mock 파일을 프로덕션 및 모노레포 내부 
     코드 베이스에 생성하지 않는다. 모든 파일 구조는 실제 동작 가능한 엔티티로만 구성한다.
  2) [Dummy 데이터 생성 금지]: 런타임 또는 소스 코드 내부에 유효하지 않거나 구조만 채우는 
     가짜 dummy 데이터를 하드코딩하여 사용하지 않는다.
  3) [Fake API 및 Fake 성공 응답 금지]: 외부 인프라가 연동되지 않았다고 해서 임시로 성공을 
     가장한 응답({ success: true } 등)을 반환하거나, 모조 API 공급자를 임시 작성하지 않는다.
  4) [Seed 데이터 배제]: 데이터의 정합성을 어지럽힐 수 있는 임의성 seed 데이터는 초기 
     MVP 구현 범위 및 운영 데이터베이스 적용 대상에서 완전히 제외한다.
  5) [외부 연동 값의 완벽한 분리]: 외부 연동(카카오 OAuth, OpenAI, Solapi, Gmail 등)에 
     필요한 모든 Key, Secret, URI 등의 민감 정보는 코드 내에 절대 하드코딩하지 않고, 
     오직 운영 환경 및 로컬 환경의 환경 변수(`.env`) 파일을 통해서만 주입받는다.
  6) [.env.example 작성 원칙]: 저장소에 버전 관리되는 `.env.example` 파일에는 실제 값은 
     물론이고, 어떠한 가짜 값(예: 'your-api-key')도 기재하지 않는다. 오직 시스템 구동에 
     필수적인 '환경 변수 Key 목록'만을 클린하게 나열하여 제공한다.
  7) [필수 환경 변수 누락 시 Fail-Fast 프로세스]: 서비스가 구동(서버 시작)될 때 필수 
     환경 변수가 하나라도 유실되거나 설정되지 않은 경우, 기본값으로 우회하지 않고 시스템이 
     즉시 명확한 런타임 오류를 발생시키며 크래시(Crash)되도록 구현한다.
  8) [미설정 상태의 명확한 표현]: 실제 공급자 정보(Provider Metadata)나 라이브러리가 
     부재하여 동작할 수 없는 미구현 기능은 성공으로 가장하지 않고, 사용자 및 에이전트에게 
     "명확한 설정 누락 오류(UNCONFIGURED)" 또는 "미설정 상태 오류"를 즉시 명시적으로 노출한다.

2. 인공지능(AI) 개발 및 활용 원칙 (AI Engineering Boundary)
--------------------------------------------------------------------------------
AI는 사용자 및 개발자의 작업을 구조화하고 가이드라인을 제공하는 보조 도구로만 제한되며, 
안전 장치를 우회하거나 보안 경계를 침해하지 못하도록 강하게 격리합니다.

[2.1 AI의 명확한 역할 경계]
  - 허용 범위: 자연어 명령의 구조화 변환(Command JSON Schema 파싱), 규칙 및 정책 초안 작성, 
    README.md 자동 서식 생성 및 템플릿 제안, 캐릭터 및 사용자 대응용 텍스트 문구 생성.
  - 절대 차단: 로컬 및 서버 시스템 파일의 임의 변경/삭제 권한 부여, 사용자의 명시적인 
    승인(Approve) 단계를 생략한 비결정론적 파일/DB 쓰기 작업의 직접 수행, 
    개인정보 및 보안 토큰의 AI 모델 직접 노출.

[2.2 검증 가드레일]
  - AI 기술(OpenAI 등)로부터 파싱되어 넘어오는 모든 데이터는 서비스 내부로 진입하기 전 
    반드시 서버 단에서 강력한 유형 검증(Zod Schema 혹은 명시적 DTO Validation)을 거쳐야 한다.
  - 외부 AI 연동 인프라가 정상 구동하지 않거나 환경 변수가 누락된 경우, 원칙 1-8에 의거해 
    가짜 성공을 유도하지 않고 즉시 사용자 화면과 백엔드 시스템에 '명확한 설정 누락'을 표기한다.

3. 모노레포 구조 및 담당 영역 분업 프로세스 (Monorepo Ownership)
--------------------------------------------------------------------------------
물리적 레포지토리 분리로 인한 공통 타입 유실 및 버전 관리 비용을 방지하기 위해 단일 
레포지토리(Monorepo) 구조를 유지하되, 폴더 소유권과 책임을 완전히 격리하여 운영합니다.

[3.1 디렉터리 소유권 경계]
  - Frontend 전용 소유: `apps/web/**` (Next.js, React, Tailwind UI, Figma UI 최적화 반영)
  - Backend 전용 소유: `apps/api/**`, `packages/database/**`, `infra/**`, `scripts/**` 
    (Express API, 인증/인가 처리, Prisma 마이그레이션, 외부 연동 비즈니스 로직 및 배포 스크립트)
  - 공동 소유 (계약 영역): `packages/shared/**`, `.env.example`, `README.md`, 기획 설계서 일체

[3.2 영역별 독립 수정 및 단독 수정 불가 제한]
  - Frontend 담당자는 화면 레이아웃 변경, 컴포넌트 고도화, 유효성 검사, 로딩/에러/빈 화면 
    상태 구현을 주도하며 백엔드 소스코드 및 데이터베이스 스키마를 단독으로 수정할 수 없다.
  - Backend 담당자는 데이터 모델링, 스케줄러, 크론잡, 보안 세션, 외부 연동 모듈을 책임지며 
    프론트엔드의 화면 마크업이나 UI 구조를 임의로 조작할 수 없다.

4. API 계약 관리 및 하위 호환성 정책 (API Contract & Backward Compatibility)
--------------------------------------------------------------------------------
두 엔지니어의 소통 비용과 런타임 통신 장애를 제거하기 위해 모든 데이터 접점은 
'공개 계약(Public Contract)'으로 제어합니다.

[4.1 API 변경 프로세스 규칙]
  - 새로운 Endpoint 설계나 데이터 변형 필요 시 아래 순서에 따라 처리한다.
    1) 백엔드 담당자가 Method, Path, Auth 여부, Request Body, Response Body, Error Code 초안 작성
    2) 프론트엔드 담당자가 사용자 화면 구성에 맞춰 필요 필드 및 상태 적합성 검토
    3) 상호 합의 도달 후 `packages/shared/src/` 아래 공통 타입 파일에 계약을 명문화
    4) 백엔드 API 실제 구현 및 프론트엔드 호출 연결 수행
    5) 전체 모노레포 빌드 및 타입 체크(`pnpm typecheck`) 통과 확인

[4.2 데이터 모델의 완전 분리]
  - 데이터베이스 내부 물리 구조인 Prisma 모델(`packages/database`)은 서버 내부 전용 구현체다.
  - 프론트엔드에 노출되는 데이터 프로토콜은 오직 `packages/shared` 내부의 API Response 타입만 
    사용하도록 엄격히 분리하여, 내부 테이블의 사소한 변경이 클라이언트 단에 직접적 장애를 
    유발하지 않도록 격리한다.

[4.3 Deprecated 정책 및 하위 호환성 유지]
  - 기존에 제공되던 shared 타입의 필드명을 바꾸거나 제거할 때, 호환성 유지 기간 없이 즉시 
    삭제하는 행위를 절대 금지한다.
  - 필드 변경 시, 구 버전 필드에 `@deprecated` 주석을 명시하고 신 버전 필드와 '동시에' 제공한다.
  - 프론트엔드 마이그레이션이 완전히 종료되고 프로덕션에 최소 1회 배포 사이클이 완료된 후, 
    "더 이상 프론트엔드에서 참조하지 않음"을 완벽히 검증한 상태에서 별도 PR을 통해 기존 
    deprecated 필드를 안전하게 제거한다.
  - 대규모의 파괴적 변경(Breaking Change)이 불가피한 경우 `ResponseV2` 형태로 독립된 새로운 
    타입을 개설하여 클라이언트의 즉각적인 오작동을 원천 봉쇄한다.

5. 개발 워크플로우 및 커밋/PR 가이드라인 (Workflow & Conventions)
--------------------------------------------------------------------------------
작은 단위를 신속하게 통합하여 장기 브랜치로 인한 머지 충돌을 예방하도록 자동화된 검증 규칙을 적용합니다.

[5.1 브랜치 명명 표준 규격]
  - `frontend/*` : `apps/web` 중심의 사용자 UI/UX 최적화 작업 브랜치
  - `backend/*`  : `apps/api` 및 데이터베이스 스키마 위주의 백엔드 로직 브랜치
  - `contract/*` : API 계약서 수정, `packages/shared` 및 `.env.example` 관련 공통 구조 변경 브랜치

[5.2 구조화된 커밋 메시지 규칙]
  - 코드의 영향 범위를 명확히 알 수 있도록 아래 접두사(Prefix)를 필수로 적용한다.
    * `feat(web):`, `fix(web):`, `refactor(web):`
    * `feat(api):`, `fix(api):`, `refactor(api):`
    * `schema(db):` (Prisma 스키마 및 마이그레이션 변경)
    * `docs:` (기획, 워크플로우, 마이그레이션 안내 문서 등)
    * `chore(env):` (환경 변수 키 구조 변경 및 설정 수정 등)

6. 데이터베이스 관리 및 인프라 배포 자동화 (Database & Deploy Automation)
--------------------------------------------------------------------------------
안전한 프로덕션 배포와 롤백의 용이성을 위해 표준 스크립트를 통한 통합 검증 절차를 강제합니다.

[6.1 데이터베이스 변경 프로세스]
  - 백엔드 주도로 `schema.prisma`를 수정한 후 반드시 마이그레이션을 생성하고, 
    `pnpm db:validate` 및 `pnpm db:generate`를 통해 도구 체인의 무결성을 상시 검증한다.
  - 스키마 레이아웃의 변경이 프론트엔드 화면 표시에 단 하나의 필드라도 영향을 미칠 경우, 
    단독 머지를 금지하고 무조건 공동 리뷰 대상으로 등록하여 검수받는다.

[6.2 표준 배포 스크립트 실행 규칙 (`scripts/deploy-main.sh`)]
  - 운영 서버에 코드를 배포할 때는 반드시 승인된 표준 통합 자동 배포 스크립트를 경유한다.
  - 해당 배포 자동화 프로세스는 내부적으로 다음 15가지 세부 파이프라인을 완전 자동 실행한다.
    1) Local working tree 상태 검증 (Dirty 파일 존재 시 배포 즉각 차단)
    2) 원격 최신 소스 코드 페치 (`git fetch origin/main`)
    3) main 브랜치 전환 상태 확약
    4) 패스트 포워드 머지 수행 (`git pull --ff-only origin main`)
    5) 무결성 잠금 설치 확인 (`pnpm install --frozen-lockfile`)
    6) 데이터베이스 무결성 검사 (`pnpm db:validate`)
    7) Prisma 클라이언트 코드 재생성 (`pnpm db:generate`)
    8) 백엔드 애플리케이션 프로덕션 빌드 (`pnpm --filter @maeari/api build`)
    9) 프론트엔드 웹 애플리케이션 프로덕션 빌드 (`pnpm --filter @maeari/web build`)
    10) 프로덕션 데이터베이스 마이그레이션 배포 (`pnpm db:deploy`)
    11) PM2 백엔드 API 서비스 환경 변수 갱신 및 무중단 재시작 (`pm2 restart maeari-api --update-env`)
    12) PM2 백엔드 스케줄러 서비스 무중단 재시작 (`pm2 restart maeari-scheduler --update-env`)
    13) PM2 프론트엔드 웹 서비스 무중단 재시작 (`pm2 restart maeari-web --update-env`)
    14) PM2 현재 프로세스 상태 영속 저장 (`pm2 save`)
    15) 런타임 헬스체크 검증 (API: `/api/health`, WEB: `/` 페이지 연속 응답 확인 후 완료)

--------------------------------------------------------------------------------
7. 최종 완료 조건 (Definition of Done) 및 품질 게이트
--------------------------------------------------------------------------------
모든 PR과 신규 기능 개발은 다음의 자동화 게이트와 계약 조건을 한 치의 오차도 없이 
충족하는 상태를 '최종 완료(DoD)'로 정의합니다.

  - [프론트엔드 변경 필수 조건]: `pnpm --filter @maeari/web typecheck` 및 빌드 오류 0건 성립
  - [백엔드 변경 필수 조건]: `pnpm --filter @maeari/api typecheck` 및 빌드 오류 0건 성립
  - [데이터베이스 변경 필수 조건]: `pnpm db:validate` 성공 및 `schema.prisma` 파일 구조적 결함 없음 입증
  - [계약 변경 필수 조건]: 단독 커밋 금지, 상호 교차 리뷰 승인 로그 남기기, 전체 통합 타입 체크 성공 처리
  - [UI 완성도 가드레일]: 정상 경로 데이터의 레이아웃 확인뿐만 아니라, 로딩 상태(Loading), 데이터가 
    없는 빈 상태(Empty), 네트워크 차단 상태(Offline), 인가 오류 상태(Error)의 UI 처리 유무 검수 완료
